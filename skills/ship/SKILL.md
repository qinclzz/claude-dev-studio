# 技能：发布检查 (/ship)

**触发时机**: backlog 所有任务状态为 `done` 后执行。前置条件：`/dev` 全部完成。

## 概述

对本次需求的全部改动进行代码审查和质量门控，输出发布结论。分两步执行，每步完成后需用户确认。

## 执行步骤

### 准备：定位需求文档

输入：`/ship [需求文件夹路径]`，例如 `/ship docs/requirements/20260528-shared-goods-search-validation`

执行：
1. 读取 `PRD.md`，提取验收标准
2. 读取 `ADR.md`，提取 API 契约和设计约束
3. 读取 `backlog.md`，确认所有任务状态为 `done`

若 backlog 中仍有 `pending` 或 `in_progress` 任务，阻塞并提示未完成任务编号。

---

### 第一步：代码审查 → REVIEW.md

调用 superpowers 的 `requesting-code-review` 原子技能，重点检查：

**正确性**
- [ ] 实现是否符合 PRD 验收标准（逐条对照）
- [ ] 实现是否符合 ADR 中的 API 契约和数据结构定义
- [ ] 边界条件和异常路径是否处理

**遗留项目专项**
- [ ] 是否调用了 `@Deprecated` 方法（禁止）
- [ ] 是否存在隐藏赋值或副作用（零信任检查）
- [ ] 是否有 N+1 查询（循环内执行数据库查询）
- [ ] 事务边界是否正确（多表更新必须在同一事务内）

**安全**
- [ ] 是否有 SQL 拼接（必须使用参数化查询）
- [ ] 外部输入是否经过校验

**代码质量**
- [ ] 是否有硬编码的魔法数、URL、配置项
- [ ] 单个方法是否超过 30 行
- [ ] 是否有重复代码可以复用

发现问题后，分级列出：

```
[阻塞] 必须修复才能发布
[建议] 本次可不修复，记录为技术债
[忽略] 遗留代码，不在本次范围
```

写入 `REVIEW.md`：

```markdown
# 代码审查：{需求标题}

> 审查时间：{date}
> 关联 PRD：./PRD.md

## 验收标准对照
| 验收标准 | 状态 | 说明 |
|---------|------|------|

## 发现问题
### [阻塞]
### [建议]

## 结论
```

**[确认节点]** 展示审查结果，等待用户选择：
- [确认，进入质量门控] → 继续第二步（仅当无阻塞问题时可选）
- [返回修复阻塞问题] → 终止，回到 `/dev`

---

### 第二步：质量门控 → GATE.md

执行最终发布前检查清单：

**功能完整性**
- [ ] PRD 所有 Must-Have 功能已实现
- [ ] 所有验收标准已通过
- [ ] backlog 所有任务状态为 `done`

**技术质量**
- [ ] 代码审查无阻塞问题
- [ ] 编译通过，无编译警告（新增部分）
- [ ] 相关测试全部通过

**发布安全**
- [ ] 无敏感信息（密码、密钥）提交到代码
- [ ] 数据库变更脚本已准备（若有表结构变更）
- [ ] 接口变更已通知下游调用方（若有）

写入 `GATE.md`：

```markdown
# 质量门控：{需求标题}

> 检查时间：{date}
> 关联 REVIEW：./REVIEW.md

## 检查清单
（逐项列出通过/未通过状态）

## 发布结论

> [通过] / [附条件通过] / [阻塞，不可发布]

## 附条件说明（若适用）
（列出可在发布后跟进的遗留问题）
```

**[确认节点]** 展示门控结果，给出三种结论：
- **[通过]**：所有检查项通过，可以合并发布
- **[附条件通过]**：存在建议级问题，可发布但需记录跟进
- **[阻塞，不可发布]**：存在阻塞问题，必须修复后重新执行 `/ship`

---

### 第三步：发布收尾（通过后）

调用 superpowers 的 `finishing-a-development-branch` 原子技能：

- 确认 branch 名称和目标合并分支
- 检查是否有未提交的改动
- 提示用户执行 PR 创建或合并操作

## 产物

```
docs/requirements/{YYYYMMDD}-{slug}/
  REVIEW.md    代码审查报告
  GATE.md      质量门控报告
```

## 成功门控

- REVIEW.md 无阻塞问题
- GATE.md 所有必选检查项通过
- 发布结论明确（通过 / 附条件通过 / 阻塞）

## 后续步骤

- 通过：合并分支，需求完成
- 阻塞：回到 `/dev` 修复问题后重新执行 `/ship`
