# 技能：需求启动 (/kickoff)

**触发时机**: 每次新需求开始时执行。前置条件：`/onboard-project` 已完成，`.claude/studio/system-map.md` 存在。

## 概述

将一句话需求描述转化为三份文档：PRD（需求说明）、ADR（架构决策）、backlog（任务清单）。三步强依赖，顺序执行，每步完成后需用户确认再进入下一步。

## 执行步骤

### 准备：创建需求文件夹

根据用户输入生成需求文件夹：

```
命名规则：docs/requirements/{YYYYMMDD}-{slug}/
slug 规则：由用户提供英文简称，Claude 只负责拼接当天日期
示例：
  用户输入：/kickoff 共享商品搜索校验需求
  Claude 询问：请提供英文简称（如 shared-goods-search）
  用户回复：shared-goods-search
  文件夹：docs/requirements/20260528-shared-goods-search/
```

创建文件夹，后续三步产物均写入此目录。同时将此路径记录在当前会话上下文，供 `/dev` 定位 backlog 使用。

---

### 第一步：需求澄清 → PRD.md

**目标**：将模糊需求转化为有验收标准的 PRD。

提问清单（逐一确认，不得跳过）：

1. 业务背景：这个需求解决什么问题？当前痛点是什么？
2. 核心功能：Must-Have 的功能点有哪些？（要求用户列举，不得推测）
3. 边界：明确不在本次范围内的功能
4. 验收标准：如何判断这个需求做完了？（要求具体可验证的条件）
5. 影响范围：已知会影响哪些模块或接口？

澄清完成后生成 PRD 草案，写入 `PRD.md`：

```markdown
# PRD：{需求标题}

> 状态：Draft
> 创建时间：{date}
> 需求路径：docs/requirements/{folder}/

## 业务背景
## 核心功能（Must-Have）
## 边界（Out of Scope）
## 验收标准
## 已知影响范围
```

**[确认节点]** 展示 PRD 草案，等待用户选择：
- [批准，进入架构设计] → 继续第二步
- [需要修改] → 重新澄清
- [取消] → 终止

---

### 第二步：影响分析 → ADR.md

**目标**：基于 PRD 和系统地图，确定改动方案和影响范围。

执行动作：

1. 读取 `.claude/studio/system-map.md`，定位 PRD 中涉及的模块
2. 对每个涉及模块，用 graphify 查询其依赖关系（上游调用方、下游依赖）
3. 识别改动风险点（高频调用路径、共享接口、数据库表）

生成 ADR 草案，写入 `ADR.md`：

```markdown
# ADR：{需求标题}

> 状态：Draft
> 关联 PRD：./PRD.md
> 创建时间：{date}

## 背景
（引用 PRD 业务背景）

## 涉及模块
| 模块 | 改动类型 | 风险等级 |
|------|---------|---------|

## 影响范围
（graphify 查询结果：上游调用方、下游依赖）

## 改动方案
（2-3个方案对比，推荐方案加粗）

## 决策
（用户选择的方案及理由）

## API 契约 / 数据结构变更
（若有接口或表结构变更，在此定义）

## 权衡与风险
```

**[确认节点]** 展示 ADR 草案，等待用户选择：
- [批准，进入任务拆解] → 继续第三步
- [需要调整方案] → 重新设计
- [取消] → 终止

---

### 第三步：任务拆解 → backlog.md

**目标**：将 ADR 中的改动方案拆解为可执行的原子任务。

拆解原则：
- 每个任务独立可执行，预估工时不超过 4 小时
- 按依赖顺序排列（数据库变更 → 后端接口 → 前端 → 测试）
- 每个任务标注所属模块和文件范围

写入 `backlog.md`：

```markdown
# Backlog：{需求标题}

> 关联 ADR：./ADR.md
> 创建时间：{date}

## 任务清单

| # | 任务描述 | 模块 | 预估 | 状态 | 依赖 |
|---|---------|------|------|------|------|
| 1 | | | | pending | - |
| 2 | | | | pending | 1 |

## 完成标准
（引用 PRD 验收标准）
```

**[确认节点]** 展示 backlog，等待用户选择：
- [批准，开始开发] → 技能结束，提示执行 `/dev`
- [需要调整任务] → 重新拆解

## 产物

```
docs/requirements/{YYYYMMDD}-{slug}/
  PRD.md
  ADR.md
  backlog.md
```

## 成功门控

- 三份文档均已生成且用户批准
- PRD 包含可验证的验收标准
- ADR 包含 graphify 查询的影响范围
- backlog 每个任务有明确的模块和文件范围

## 后续步骤

按 backlog 任务顺序，逐个执行 `/dev [任务编号]`。
