# 技能：单任务开发 (/dev)

**触发时机**: 执行 backlog 中的单个任务。前置条件：`/kickoff` 已完成，`backlog.md` 存在。

## 概述

以零信任原则阅读遗留代码，TDD 驱动实现单个 backlog 任务，完成后更新任务状态并提交 commit。

## 执行步骤

### 第一步：读取任务上下文

输入：`/dev [任务编号]`，例如 `/dev 3`

一个会话只处理一个需求。需求路径从以下来源获取（优先级从高到低）：
1. 当前会话中 `/kickoff` 执行时记录的需求文件夹路径
2. 若会话中未执行过 `/kickoff`，询问用户：`请提供需求文件夹路径（如 docs/requirements/20260528-shared-goods-search/）`

确认路径后执行：
1. 读取 `{需求路径}/backlog.md`，定位指定任务，确认状态为 `pending`
2. 读取 `{需求路径}/ADR.md`，提取该任务相关的设计约束和 API 契约
3. 将任务状态更新为 `in_progress`

若任务有前置依赖（`依赖` 列不为 `-`），检查依赖任务状态是否为 `done`，未完成则阻塞并提示。

---

### 第二步：零信任代码阅读

**强制执行，不得跳过。**

遵循全局 CLAUDE.md 中的零信任阅读原则：

1. **定位终点**：先找目标变量/结果最后一次被赋值的位置
2. **反向溯源**：从赋值点逐层向上追踪调用链
3. **确认实现**：必须定位到具体 `ServiceImpl`，不得停留在 Interface 层
4. **副作用检索**：主动搜索 `set`、`put`、`update`、`add` 等状态变更操作

输出阅读报告（内部使用，不写文件）：
```
[调用栈轨迹] 类A:行N → 类B:行M → 类C:行K
[关键判定点] 文件路径:行号，条件内容
[状态变更点] 变量在何处被修改，来源是什么
[核心计算式] 最终结果的计算逻辑
```

阅读完成后，向用户展示改造设计方案，**等待确认后再动手**。

---

### 第三步：TDD 实现

调用 superpowers 的 `test-driven-development` 原子技能：

1. 先写测试（基于 ADR 中的 API 契约和验收标准）
2. 运行测试，确认红灯
3. 实现最小可用代码，使测试通过
4. 重构，保持测试绿灯

遗留项目特殊规则：
- 若现有代码无测试框架，先确认用户是否需要补充，不强制
- 集成测试必须连接真实数据库，禁止 Mock 数据库层
- 外部 HTTP 接口调用可以 Mock

---

### 第四步：遇到 Bug 时

调用 superpowers 的 `systematic-debugging` 原子技能：

1. 复现问题，记录现象
2. 缩小范围，定位根因（附文件路径:行号）
3. 修复，验证修复不引入新问题

---

### 第五步：提交前验证

调用 superpowers 的 `verification-before-completion` 原子技能：

- [ ] 代码编译通过
- [ ] 相关测试全部通过
- [ ] 无新增的 `@Deprecated` 调用
- [ ] 无硬编码的魔法数、URL、配置项
- [ ] 符合 ADR 中定义的 API 契约

验证通过后，向用户展示改动摘要，**等待用户确认后再执行 git commit**。

---

### 第六步：git commit

commit message 规则：
1. 从 `git log` 读取本次需求已有的 commit，提取前缀
2. 若无已有 commit，提示用户提供前缀
3. 格式：`{前缀} {任务描述}`

执行 commit 后，将 `backlog.md` 中该任务状态更新为 `done`。

## 产物

- 代码文件变更
- git commit
- `backlog.md` 任务状态更新为 `done`

## 成功门控

- 用户确认了设计方案后才开始编码
- 测试通过（若有测试）
- 用户确认了改动摘要后才执行 commit
- commit message 包含需求前缀
- backlog 任务状态已更新

## 后续步骤

所有任务完成后，执行 `/ship [需求文件夹路径]` 进行发布检查。
